基于 Sitemap 的 Nature 期刊爬虫项目总结

2026-06-30
  • python
  • python
  • 爬虫
  • 自动化

这个项目是一个面向 Nature 官网期刊页面的批量文章发现与 PDF 下载脚本。它不从列表页逐页翻页，而是基于 Nature 的 sitemap 结构定位期刊、年份和月份对应的文章 XML，再对文章页提取元数据和 PDF 下载链接，最后把 PDF 与文章信息按期刊、年份、月份归档保存。

项目目标

项目的目标是自动化完成以下工作：

  1. 根据期刊短码选择目标 Nature 期刊。
  2. 从 Nature 全站 sitemap 中筛选该期刊指定年份范围内的文章 sitemap。
  3. 从每个月份的文章 sitemap 中发现文章 URL。
  4. 请求文章详情页，提取标题、DOI、日期、作者、文章类别、摘要、PDF 链接等信息。
  5. 下载可访问的 PDF 文件，并按 MD5 命名，避免重复保存。
  6. 将文章元数据写入 CSV，支持中断后继续运行。

这种方式相比直接爬取页面列表更稳定，因为 sitemap 天然适合做全量 URL 发现，也能更方便地按年份和月份控制范围。

整体流程

主流程集中在 nature.py 的 main() 函数中，大致可以分成六步。

第一步，解析命令行参数。脚本要求至少传入一个期刊短码，并可选传入起始年份和结束年份。如果年份不合法，或者起始年份大于结束年份，程序会直接提示并退出。

第二步，根据短码查找期刊全名。nature_list.py 中维护了期刊名称和 Nature URL，主程序会从 URL 末尾提取短码，并构造短码到期刊名称的映射。

第三步，准备下载目录和 CSV 文件。下载根目录是 downloads/，每个期刊会创建独立子目录，CSV 文件保存在对应期刊目录下。

第四步，读取已有 CSV，实现断点续爬。如果 CSV 中已经存在某篇文章的 URL，后续扫描 sitemap 时会跳过它，避免重复下载。

第五步，从本地 nature_sitemap.xml 中筛选目标期刊的文章 sitemap。如果本地 sitemap 不存在，脚本会先请求 https://www.nature.com/sitemap.xml 并缓存到本地。

第六步，边扫描 sitemap 边提交下载任务。每发现一个新的文章 URL，就提交到线程池中处理，同时主线程会定期回收已完成任务，把成功结果写入 CSV。

每个子 sitemap 中包含若干 <url> 节点，脚本只保留 URL 中包含 /articles/ 的文章页面，排除其他非文章链接。

元数据提取

每篇文章由 process_article() 处理。它首先调用 get_article_metadata() 请求文章详情页，并从页面的 meta 标签中提取字段。

提取内容包括：

  • 标题：dc.title 或 citation_title
  • DOI：citation_doi 或 dc.identifier
  • 发布日期：dc.date、citation_date 或 prism.publicationDate
  • 作者：所有 dc.creator
  • 文章类别：prism.section
  • 摘要：优先读取 dc.description
  • PDF 链接：寻找 data-track-action="download pdf" 的链接
  • 开放访问状态：检测 data-test="open-access" 元素

如果页面没有提供摘要 meta 标签，脚本会尝试在正文中寻找标题为 Abstract 的区域，再拼接其后的内容作为摘要。

PDF 下载与去重

PDF 下载由 download_pdf() 完成。这个函数有几个实用设计：

  1. 使用 stream=True 流式下载，避免一次性把大 PDF 读入内存。
  2. 先读取文件前 4 个字节，检查是否以 %PDF 开头，避免把登录页、错误页或反爬页面保存成 PDF。
  3. 下载过程中同步计算 MD5。
  4. 先写入临时文件，下载完成后再改名为 {md5}.pdf。
  5. 如果同一个 MD5 的文件已经存在，就删除临时文件并复用已有 PDF。

并发模型

项目使用 ThreadPoolExecutor，默认线程数为 5：

主线程负责扫描 sitemap 和提交任务，工作线程负责请求文章页、提取元数据、下载 PDF。程序不是先收集完所有 URL 再统一下载，而是“边发现边下载”。

这个设计的好处是：

  • 不需要在内存中保存所有文章 URL。
  • 可以尽早开始下载，提高整体吞吐。
  • 日志能实时显示当前提交、下载中、已完成和失败数量。

由于多个线程会共享计数器、CSV 文件和 URL 集合，项目使用了多个锁：

  • csv_lock：保护 CSV 写入。
  • counter_lock：保护下载成功和失败计数。
  • submitted_urls_lock：保护已提交 URL 集合。

断点续爬

断点续爬依赖 CSV 中的 URL 字段。程序启动时，如果发现目标期刊的 CSV 已存在，就读取其中所有 URL，加入 existing_urls 集合。

后续扫描 sitemap 时，如果文章 URL 已在 existing_urls 中，就直接跳过。这能避免重复下载已完成文章，也适合按年份多次补跑。

反爬与访问限制处理

脚本在请求文章页时设置了浏览器风格的 User-Agent、Accept 和 Accept-Language。同时还预留了 COOKIES 配置，用于带上会话信息。

如果文章页返回内容过短并包含 _fs-ch-，脚本会认为触发了反爬验证，并抛出提示：

PDF 下载阶段，如果拿到的内容不是 %PDF 开头，程序会认为 PDF 无权限或响应内容不是 PDF，不会保存错误文件。

项目优点

这个项目比较实用的地方在于：

  • 基于 sitemap 做文章发现，覆盖范围清晰。
  • 支持期刊短码和年份范围，控制粒度比较好。
  • 支持断点续爬，适合长时间批量任务。
  • 使用 MD5 命名 PDF，天然具备文件去重能力。
  • PDF 有魔数校验，能减少错误文件污染。
  • 元数据和 PDF 分开保存，便于后续检索、清洗或入库。
  • 线程池并发下载，复杂度不高但效率明显优于单线程。

可以改进的地方

项目当前已经能完成核心任务，但还有一些可以继续优化的方向。

第一，Cookie 不适合硬编码在源码中。更好的方式是从环境变量、配置文件或命令行参数中读取，避免泄露个人会话信息。

第二，年份默认值写死为 2000-2026。如果长期使用，可以把结束年份默认设为当前年份，减少每年维护源码的成本。

第三，CSV 中的摘要使用 Base64 虽然稳妥，但降低了人工阅读体验。后续可以同时保存原始摘要字段和编码字段，或者改用 JSONL、SQLite 等更适合结构化文本的格式。

第四，失败记录目前主要输出在日志中。对于大规模任务，可以增加失败 CSV，把失败 URL、原因、年份、月份保存下来，方便单独重试。

第五，下载权限和 OA 状态可以进一步细分。现在非 OA 但可下载的文章会标记为 ACCESS，如果需要做数据统计，可以额外保存访问状态字段。

第六，当前线程数固定为 5。可以把它做成命令行参数，方便根据网络环境、目标站点压力和本机性能调整。

总结

这个项目是一个结构清晰的 Nature 期刊 PDF 批量下载工具。它选择了 sitemap 作为入口，避开了列表页翻页的不稳定性；通过文章页 meta 标签提取元数据，保证信息来源明确；再结合流式下载、MD5 去重、CSV 断点续爬和线程池并发，使脚本具备了较好的可用性。

从工程角度看，它不是一个复杂系统，但已经覆盖了爬虫任务中最关键的几个环节：URL 发现、页面解析、文件下载、去重、进度记录和失败处理。后续如果要继续扩展，可以围绕配置化、失败重试、数据存储格式和任务调度做增强。

LoLSearch —— 基于 LCU API 的隐藏战绩查询工具

2026-06-22
  • python
  • FastAPI
  • LCU API

项目概述

LoLSearch 是一个能够查询英雄联盟任意玩家隐藏战绩的 Web 应用。当玩家在 League 客户端中将生涯设为”隐藏”后，OP.GG、U.GG 等第三方网站将无法获取其战绩——而 LoLSearch 利用 LCU（League Client Update）内部 API 绕过了这一限制。

谎言不会伤人，真相才是快刀。

项目后端基于 FastAPI，前端使用原生 HTML/CSS + jQuery + Tailwind CSS，以本地服务的方式运行。输入 召唤师名称#tagLine，即可获取该玩家的完整资料、段位和最近对局详情。

核心原理：为什么 LCU API 能破壁？

公开 API vs 内部 API

Riot 提供了两套 API：

Riot Developer API（公开） LCU API（内部）
使用方 第三方网站（OP.GG、U.GG） League 客户端本身
隐私策略 尊重隐私设置 → 隐藏战绩返回 403 不检查隐私设置 → 始终返回数据
认证方式 API Key 申请 本地 HTTPS + Riot 签名证书
访问范围 受 Rate Limit、权限分级限制 客户端能看的全能查

关键洞察： LCU API 是 League 客户端自己调用后端数据的接口。客户端需要渲染”对局记录”界面时，必然能拉到全部数据——包括隐私账号的。这个接口没有做服务端的隐私过滤。

项目中的实现架构

本项目通过一个 LCU 代理 暴露这些内部接口。代理连接本地运行的 League 客户端（LCU 进程），将内部 HTTPS/WSS 请求转换为标准 HTTP REST API：

LCU 代理是外部组件（不在本项目中），它的作用是将 LCU 的本地认证连接桥接为标准 HTTP。社区常见实现包括 lcu-driver、riot-client-api 等。

FastAPI 入口 (main.py)

整个后端只有一个接口：

数据流清晰简洁——7 个步骤完成一次查询：

  1. get_puuid() —— 将 name#tagLine 解析为 Riot 的全局唯一 ID（puuid）
  2. find_summoners_by_puuid() —— 获取召唤师基本信息
  3. get_summoner_background_skin() —— 获取个人资料背景皮肤图
  4. get_match_history() —— 拉取最近 9 场对局
  5. get_ranked_stats() —— 拉取排位数据
  6. 对每场对局调用 match_detail() —— 补充英雄、符文、KDA、装备等详情
  7. 返回统一 JSON 结构

精妙设计 ①：对局详情的数据拼接

这里利用了 **Python 3.9+ 的字典合并运算符 |**——将”玩家身份”字典和”游戏数据”字典优雅合并，避免了繁琐的逐个字段赋值。这是 Python 现代语法的精妙运用。

精妙设计 ②：符文图标映射的 O(1) 构建

DDragon 的符文 JSON 是三层嵌套结构：符文树 → 插槽组 → 具体符文。此方法将其扁平化为单一的 {id: icon_path} 字典，使得在渲染对局时任何符文的图标查询都是 O(1) 的哈希查找，而不是每次都遍历嵌套结构。

精妙设计 ③：动态版本更新

每次启动时，应用会从 ddragon.leagueoflegends.com/api/versions.json 获取最新游戏版本号，与本地 libs/GameVersion 文件中记录的版本对比。如果不一致，自动下载最新符文数据。这样符文图标永远不会因版本更新而 404。

精妙设计 ④：背景皮肤的反查链路

这段代码揭示了一个有趣的事实：LCU 只返回 backgroundSkinId（如 222016），而不直接给图片 URL。要获得实际的皮肤图片，需要：

  1. 取 skinId 的前两位作为英雄 ID（22）
  2. 请求腾讯 CDN 的 hero/22.js 获取该英雄的皮肤列表
  3. 在皮肤列表中匹配完整 skinId，提取 centerImg 字段

这是专为中国区（腾讯代理）定制的 CDN 链路，腾讯 game.gtimg.cn 的 JS 文件实质上是静态 JSON 数据，但以 .js 后缀提供。

精妙设计 ⑤：英雄 ID 懒加载缓存

  • 懒加载：只有第一次查询时才请求 DDragon 的全量英雄数据（~2MB），之后命中内存缓存
  • 索引优化：DDragon 原始数据以英雄英文名为 key（如 "Aatrox"），但 LCU 返回的是数字 ID（如 266）。这里反过来以 key 字段（数字 ID）为索引重建了字典
  • **双下划线前缀 __champion_by_id**：Python 的 name mangling，使其成为私有属性，防止外部意外修改

时间解析工具

LCU 返回的 gameCreationDate 是 ISO 8601 UTC 格式（如 2026-06-15T14:12:03.967Z）。这个函数将其转换为北京时间，同时把对局秒数格式化为中文的”X分Y秒”——简洁但考虑了本地化细节。

六项 LCU API 端点全解析

端点 用途 返回数据
lol-summoner/v1/alias/lookup name#tagLine → puuid {"puuid": "xxx"}
lol-summoner/v2/summoners/puuid/{puuid} puuid → 召唤师信息 名称、图标、等级、隐私状态
lol-summoner/v1/summoner-profile/?puuid={puuid} 个人资料背景 backgroundSkinId
lol-match-history/v1/products/lol/{puuid}/matches 对局列表 最近 9 场对局摘要
lol-match-history/v1/games/{gameId} 单局详情 10 人完整数据
lol-ranked/v1/ranked-stats/{puuid} 排位统计 单双排、灵活、TFT 段位

这些端点都不是 Riot 公开开发者 API 的一部分——它们只存在于 LCU 内部，正常手段根本无法从外部直接访问。

CDN 资源整合策略

本项目整合了多个 CDN 来源，每类资源选择了最优的获取路径：

资源类型 CDN 来源 说明
英雄数据、符文、版本号 ddragon.leagueoflegends.com Riot 官方 Data Dragon
英雄头像、装备图标、召唤师技能 game.gtimg.cn 腾讯游戏 CDN（国服高速）
段位图标、召唤师头像 wegame.gtimg.com WeGame CDN
前端依赖（Notyf、Tailwind） cdn.jsdelivr.net 通用公共 CDN

为什么混用 Riot 和腾讯 CDN？ 因为中国区由腾讯代理运营，game.gtimg.cn 在中国大陆的访问速度远优于 Riot 官方 CDN。对于面向中国用户的工具来说，这是重要的性能优化。

总结与展望

项目亮点

  1. 核心洞察——发现并利用 LCU API 不检查隐私设置的特性，实现了隐藏战绩查询
  2. 代码简洁——176 行后端 + 477 行前端，轻量高效
  3. 细节打磨——懒加载缓存、版本自动更新、O(1) 符文查询、防除零 KDA 计算
  4. 中国区适配——腾讯 CDN 混用、中文段位映射、东八区时间转换
  5. 交互闭环——点击对局中任意玩家即可连锁查询，形成无限探索的体验

可改进方向

  • 将硬编码的代理地址、CORS 来源改为环境变量或配置文件
  • 补充 requirements.txt 方便一键安装
  • FastAPI 增加静态文件服务，统一托管前端，无需额外 Live Server
  • 增加异常处理——某个 CDN 挂了不影响其他数据渲染
  • 补充单元测试覆盖工具类和数据格式化函数
  • 将 search.js 中的 HTML 字符串拼接改为模板引擎，提升可维护性
